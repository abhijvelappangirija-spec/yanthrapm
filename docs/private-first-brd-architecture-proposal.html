Private-First BRD Architecture Proposal

Version: 1.0

Date: March 23, 2026

Audience: Abhij

Executive Summary

The current BRD generation flow is useful for prototyping, but it is not sufficient for sensitive or enterprise-grade usage. Today, user content is sent directly to an external AI provider and the returned BRD is stored as HTML with limited validation. This increases confidentiality risk and allows hallucinated or weakly grounded content to reach the final document.

The proposed solution is a private-first hybrid architecture. Sensitive inputs remain in a private model path. A local or private model extracts facts, constraints, assumptions, and open questions into structured JSON before writing a BRD. If fresh industry information is needed, a separate retrieval sidecar fetches only generic, sanitized, non-sensitive information from approved sources. The final BRD is generated only after evidence extraction, validation, and output safety controls.

This does not create perfect security or perfect correctness. No system can claim that honestly. It does create a much stronger and more defensible architecture for client-sensitive BRD generation.

1. Why Change Is Needed

The current implementation is strong enough for experimentation, but not for confidential or client-facing use. The main issues are:

  1. Sensitive business ideas and requirements can leave the application boundary.
  2. The BRD is generated in a single pass, which makes hallucination harder to control.
  3. The current prompts are generic and do not consistently produce architect-grade requirements.
  4. The system trusts AI output and client-supplied identifiers more than it should.
  5. The workflow does not clearly separate facts, assumptions, open questions, and current external references.

2. Current Architecture

The current user flow is straightforward:

User Input / File Upload | v BRD API Endpoint | v External AI Provider | v HTML BRD Response | v Store and Render

Current strengths

  • Fast to build.
  • Low operational overhead.
  • Good for demos and rapid experimentation.

Current limitations

  • Direct external handling of user content creates confidentiality concerns.
  • Single-pass generation makes it easier for the model to invent unsupported details.
  • There is limited separation between data extraction, reasoning, document writing, and validation.
  • Current rendering and storage need stronger safety controls.

3. Proposed Future Architecture

The target design is private-first, evidence-based, and policy-driven.

User Input / Uploaded File | v Input Validation + Sensitivity Classification | v Private Evidence Extraction (Ollama / private model) | +------------------------------+ | | | if policy allows and needed | v | Sanitized External Retrieval Sidecar | | | +------------> Curated Facts --+ | v Requirement Structuring + Assumptions Register | v Private BRD Writer | v Validation Layer - supported claims - section completeness - schema checks - HTML sanitization | v Storage + Review + Export

What this changes

  • Raw sensitive input stays in the private path by default.
  • Current or latest information is fetched separately from the BRD-writing path.
  • The BRD is based on structured evidence instead of one large prompt.
  • Validation becomes a first-class step rather than an optional improvement.

4. How the New Plan Works

  1. The user submits notes, files, or source material.
  2. The system validates the input and classifies its sensitivity.
  3. A private model extracts structured facts, goals, roles, constraints, assumptions, and unknowns.
  4. If the BRD needs current industry or standards information, a retrieval sidecar issues generic, sanitized queries to approved sources.
  5. The system merges private evidence and approved current facts into a normalized requirement package.
  6. A private BRD writer creates the first draft from structured input only.
  7. A validator checks missing sections, unsupported claims, contradictions, and unsafe HTML.
  8. The BRD is stored, reviewed, and exported only after passing the quality and safety gates.

5. What Makes This More Like a Technical Architect or Manager

A strong technical manager or architect does not produce a BRD by simply rephrasing raw notes. A professional process usually includes:

  1. Separating facts from assumptions.
  2. Making scope and out-of-scope explicit.
  3. Capturing dependencies, constraints, and risks.
  4. Defining measurable non-functional requirements.
  5. Calling out open questions rather than hiding uncertainty.
  6. Using current standards and domain references where needed.

The proposed design mirrors that process in the system architecture.

7. Advantages

Security advantages

  • Major reduction in sensitive data exposure to third-party AI providers.
  • Clear policy on what can and cannot leave the environment.
  • Better fit for confidential client and strategic project work.
  • Stronger foundation for authentication, row-level security, and auditability.

Document quality advantages

  • More practical BRDs with explicit constraints, risks, and assumptions.
  • Better non-functional requirements, security requirements, and acceptance criteria.
  • Reduced hallucination due to evidence-first generation.
  • Improved trust for product, engineering, and client review.

Delivery advantages

  • Reusable staged architecture for future features.
  • Easier to evolve models and prompts without changing the whole system.
  • Supports later approval workflows and quality scoring.

8. Disadvantages and Tradeoffs

  • The architecture is more complex than a direct external API call.
  • Private inference requires infrastructure planning and model operations.
  • Local models may be weaker than frontier hosted models in some writing tasks.
  • There is more up-front engineering work and more policy design to get right.
  • Performance tuning becomes more important when local inference is added.
Important limitation: this architecture is significantly more secure, but not perfectly secure. No serious production system should promise 100 percent security or 100 percent correctness.

9. Feasibility Assessment

Technical feasibilityHigh

  • The current project already has API boundaries where orchestration can be introduced.
  • The existing AI integration can be refactored into a provider-based pipeline.
  • Private inference with Ollama is achievable without a full rewrite.

Security feasibilityHigh

  • Stronger access controls, output sanitization, and policy routing are standard engineering practices.
  • The main work is disciplined implementation, not theoretical invention.

Operational feasibilityMedium

  • Private inference introduces infrastructure responsibility.
  • Latency and concurrency expectations need to be defined.

Commercial feasibilityMedium to High

  • The added cost is justifiable if client confidentiality and higher document quality are important differentiators.

10. Recommended Working Model

The recommended operating model is controlled hybrid with private-first defaults.

  1. All client-specific and project-specific input starts in the private path.
  2. External retrieval is disabled by default for highly sensitive work.
  3. When external freshness is needed, only generic sanitized lookups are allowed.
  4. The final BRD is always generated in the private path.
  5. If the private path is unavailable for confidential work, the request should fail closed rather than falling back to an external provider.

Phase 2 Provider abstraction and model routing Clean path to hybrid operation

Phase 4 Controlled external retrieval sidecar Current information without direct exposure of sensitive input
Phase 5 Quality scoring, review gates, governance Enterprise-ready reviewable workflow

12. How We Will Work

The implementation should be phased and disciplined:

  1. Secure the existing foundation first.
  2. Refactor the AI integration into independent stages.
  3. Introduce private inference before adding external retrieval.
  4. Add evidence extraction and validation before changing final BRD style.
  5. Measure output quality and security posture at each phase.

Suggested ownership

  • Product / BA: define required BRD template and review assumptions.
  • Engineering Lead / Architect: define policies, target flow, and technical acceptance rules.
  • Backend Engineer: implement routing, validators, storage, and orchestration.
  • Security / Platform: enforce outbound controls, identity, and observability.
  • QA: test edge cases, prompt injection resistance, and unsupported claims.

13. Main Risks and Mitigations

Risk Description Mitigation
Indirect leakage Poorly sanitized external queries may reveal sensitive details Use fixed query templates, outbound filtering, and audit logs
Model overconfidence Private model can still invent unsupported claims Use evidence-first generation and supported-claims validation
Operational overhead Local inference adds infra and monitoring needs Roll out in phases with observability from the start
Performance constraints Private inference may increase latency Right-size models, batch work, cache results, and use async flows where needed
Weak source quality External retrieval may return stale or low-quality references Restrict to approved sources and record citation date

14. Success Criteria

  1. Sensitive BRD inputs stay in the private inference path by default.
  2. Unauthorized users cannot access or modify other users' records.
  3. Generated BRDs consistently include scope, constraints, dependencies, and measurable success criteria.
  4. Unsupported claims are flagged before final save.
  5. Latest or current information is introduced only through controlled retrieval with traceability.
  6. The output quality is closer to an architect-reviewed draft than a one-shot AI summary.

15. Recommendation

The recommended direction is to move from direct external BRD generation to a private-first hybrid architecture. This is the best balance of confidentiality, quality, practicality, and enterprise readiness for the product.

The immediate next step should be Phase 1 and Phase 2:

  1. security foundation,
  2. provider abstraction,
  3. and output sanitization.

Once that baseline is complete, the evidence-first pipeline can be added with lower implementation risk and stronger quality control.

Final note: the goal is not to claim impossible perfection. The goal is to produce a significantly more secure, more practical, and more reviewable BRD workflow that stakeholders can trust.
