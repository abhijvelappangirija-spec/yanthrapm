Step-by-step: how to generate a BRD (complete user flow)

  1. Environment: Copy .env.example to .env.local. Set Supabase (NEXT_PUBLIC_SUPABASE_URL, NEXT_PUBLIC_SUPABASE_ANON_KEY), and AI keys/routing (see environment table). Restart the dev server after any change.
  2. Start the app: Run npm install then npm run dev. Open http://localhost:3000 (or your port).
  3. Sign in: Go to /auth, sign in with Supabase. Protected pages require a session; APIs receive the token via fetchWithAuth.
  4. Open BRD Generator: From the dashboard, open the BRD Generator page.
  5. Input: Either upload a supported file (txt, md, doc, etc.) or paste requirements under Or Enter Text Manually. Optionally pick a project for sprint defaults.
  6. Options:
    • Use dummy data only — skips real Perplexity/Ollama calls (good for UI-only testing).
    • Sensitive input: require private/local AI processing only — forces private routing when policy applies (typically Ollama); do not use with content that triggers “restricted” classification unless you intend private processing.
  7. Generate: Click Generate BRD. The client calls POST /api/generate-brd with JSON { content, useDummyData, requirePrivateProcessing }.
  8. Review panels: Read AI routing, Controlled retrieval, and BRD quality / governance sections below.
  9. Iterate: Use Edit on the BRD body if needed; continue to technical context and sprint planning when ready.
  10. Export: Use Download BRD (PDF) when available.

Controlled / external retrieval — full detail

Purpose: One optional Perplexity call with search restricted to approved domains, returning structured facts (and filtered citations). Those facts are passed into the composition step as reference material only — not as confirmed customer requirements.

  • Enable: AI_RETRIEVAL_ENABLED=true in .env.local. Restart npm run dev.
  • Domains: AI_RETRIEVAL_APPROVED_DOMAINS (comma-separated). If unset, defaults include e.g. owasp.org, nist.gov, cisa.gov, ietf.org, developer.atlassian.com, supabase.com, docs.ollama.com (see lib/ai/retrieval-policy.ts).
  • Use cases: AI_RETRIEVAL_ALLOWED_USE_CASES — e.g. industry-standards, compliance-controls, technology-docs, integration-docs. The implementation picks the first allowed case in priority order (lib/perplexity.ts).
  • Perplexity-only on BRD: The retrieval HTTP call lives in the Perplexity BRD path. Set AI_BRD_PROVIDER=perplexity to test it on Generate BRD.
  • Composition vs retrieval: Evidence extraction and final BRD composition use disable_search on Perplexity so general browsing is not mixed in; only the dedicated pass uses domain-filtered search when enabled.
  • Sample input: docs/sample-brd-input-controlled-retrieval.txt (avoid emails/phones in text so preflight does not block).
Why the UI still says “Not used”: Any of: retrieval disabled in env; input restricted without AI_RETRIEVAL_ALLOW_RESTRICTED; BRD provider is Ollama; preflight failed (query too short, sensitive pattern); Perplexity returned no parseable facts JSON.

Governance and validation

  • Evidence-first: The model fills a structured BrdEvidence object before writing prose, to reduce invention.
  • Post-compose validation: validateBrdHtml checks required section titles, grounding of claims vs evidence, and produces a score.
  • Fallback HTML: If validation fails strict thresholds, finalizeBrdHtml may replace the model HTML with renderBrdHtmlFromEvidence (template) — you may see generic placeholder lines where evidence fields were empty.
  • Prompt package version: Bumped when prompts change (lib/ai/brd-pipeline-version.ts); shown in UI for audit.
  • Persistence: Optional Supabase columns can store governance and retrieval snapshots; see migration 202603260001_brds_governance_audit.sql and docs/supabase-rls-rollout.md.

How the system works (concise overview)

  • Identity: Supabase Auth; server resolves actor from JWT; actor-scoped Supabase for RLS.
  • BRD: Classify → policies → extract evidence → optional controlled retrieval (Perplexity) → compose HTML → validate → sanitize → governance → save.
  • Sprint plan: Similar provider layer with sprint-specific prompts and routes.

Local setup (short checklist)

  1. .env.local from .env.example with Supabase + AI variables.
  2. npm run dev after env changes.
  3. Sign in at /auth.
  4. For retrieval testing: AI_RETRIEVAL_ENABLED=true and AI_BRD_PROVIDER=perplexity.
  5. npm run verify before release (lint, tests, tsc, build).

Resolved product decisions (summary)

  • Authentication: Supabase Auth + server actor resolution; no client userId.
  • Private provider default: Ollama; Perplexity cannot be configured as private provider.
  • Sensitive routing can fail closed (503) if private provider missing; AI_EXTERNAL_DISABLED optional.

Deployment and operations

  1. Apply RLS and optional migrations in target Supabase (docs/supabase-rls-rollout.md).
  2. Run checklists in that document.
  3. Enable GitHub Actions for npm run verify on the remote repo.

Verification checklist

  1. npm run verify
  2. Auth session and /api/auth/actor
  3. BRD generate (text + file); governance + retrieval panels
  4. Private routing / Ollama when configured
  5. Supabase persistence when migrations applied
